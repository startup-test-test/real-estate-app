収益還元法（直接還元法）に基づく不動産投資シミュレーションエンジンの開発に向けた技術的・実務的調査報告書
1. エグゼクティブ・サマリーと本報告書の目的
本報告書は、日本国内の不動産市場において、投資家および金融機関が利用するに足る高精度な「収益還元法（直接還元法）」シミュレーターを開発するための要件定義、計算ロジック、およびデータソースを網羅的に調査・分析したものである。2025年から2026年にかけての金利上昇局面において、従来の簡易的な利回り計算では投資判断の誤りを招くリスクが高まっている。そのため、本報告書では、単なる表面的な計算式の羅列にとどまらず、不動産鑑定評価基準に基づく厳密な定義、金融機関の融資審査で使用される「ストレス金利」を用いた評価手法、さらには既存競合ツールの機能分析に基づく差別化戦略までを詳細に提示する。
特に、日本の不動産市場特有の商慣習である「積算評価（原価法）」と「収益評価」の二重構造（Dual Valuation）を理解し、それをシミュレーターのロジックにどのように落とし込むかが開発の成否を分ける鍵となる。本調査は、国土交通省の公式基準、日本不動産研究所の投資家調査、J-REITの開示データ、および主要銀行の融資姿勢に関する最新の知見を統合し、プロフェッショナル仕様の評価エンジン構築のための基礎資料として作成された。
2. 収益還元法の理論的枠組みと市場における位置づけ
不動産価格の算定には、理論的に「費用性」「市場性」「収益性」の三つの側面からアプローチする手法が存在する。それぞれ原価法、取引事例比較法、収益還元法に対応するが、投資用不動産の評価において最も核心となるのが収益還元法である。シミュレーター開発においては、この理論的背景をユーザーインターフェースや計算ロジックに正しく反映させる必要がある。
2.1 収益還元法の定義と法的根拠
収益還元法（Income Approach）とは、対象不動産が将来生み出すであろうと期待される純収益の現在価値の総和を求めることにより、対象不動産の試算価格（収益価格）を求める手法である。日本の「不動産鑑定評価基準」において、投資用不動産の評価にあたっては、この手法の適用が原則として義務付けられている 1。
この手法は、さらに以下の二つの方法に細分化される。
直接還元法（Direct Capitalization Method）: 一期間（通常は1年間）の純収益を還元利回り（キャップレート）で除して価格を求める方法。
DCF法（Discounted Cash Flow Method）: 連続する複数の期間に発生する純収益と、保有期間終了時の売却価格（復帰価格）を、それぞれの発生時期に応じて現在価値に割り引き、それらを合計する方法。
2.1.1 直接還元法の計算式と本質
本シミュレーターの中核となる直接還元法の基本式は極めてシンプルに見えるが、その構成要素には深い専門的判断が求められる。

$$P = \frac{NOI}{R}$$
ここで、
$P$ （Price）: 収益価格
$NOI$ （Net Operating Income）: 償却前営業純収益（またはNCF: Net Cash Flow）
$R$ （Cap Rate）: 還元利回り
定義上、直接還元法は「将来の収益が永続的に、あるいは安定的に続く」という仮定、または「将来の変動予測をすべて還元利回り $R$ の中に織り込む」という前提に基づいている 2。したがって、シミュレーターにおいては、ユーザーが入力する「純収益」が一時的なものではなく「標準化された（Stabilized）」数値であることを担保する入力支援機能が不可欠となる。
2.2 不動産鑑定評価基準における位置づけ
国土交通省が定める不動産鑑定評価基準において、直接還元法は「対象不動産からの収益が安定している場合」や「データ収集の制約等によりDCF法の適用が困難な場合」に有用とされている。一方で、証券化対象不動産（J-REITや私募ファンド等）の評価においてはDCF法と直接還元法の併用が義務付けられている 4。
しかし、一般的な個人投資家や中小企業がアパートやマンション一棟を購入・売却する実務の現場では、DCF法に必要な詳細な将来予測（10年後の空室率や修繕費の予測）を行うことは困難であり、また市場の相場観も「表面利回り」や「ネット利回り（直接還元法ベース）」で語られることが圧倒的に多い。そのため、一般向けのシミュレーターとしては、DCF法は「上級者向けオプション」として実装し、メインの評価エンジンは直接還元法で構築することが市場ニーズに合致する 6。

2.4 積算評価（原価法）との決定的差異と併用の必要性
日本の不動産投資市場における最大の特徴は、金融機関の融資審査において「積算評価（Cost Approach）」が依然として極めて強い影響力を持っている点である 10。
積算評価（担保価値）: 土地の相続税路線価と建物の再調達原価に基づき、「今この建物を再築し、土地を取得したらいくらかかるか（そして古くなった分を減価）」を計算する。これは「銀行が貸し倒れた時にいくらで処分できるか」という安全性の指標である。
収益評価（事業価値）: その物件がいくら稼ぐかに基づく評価。
多くのWebシミュレーターは収益還元法のみを提供しているが、実務では「収益価格は高い（利回りが良い）が、積算価格が低いため銀行融資が出ない」というケースが頻発する 12。逆に「積算価格は高いが、収益性が低くキャッシュフローが回らない」物件もある。したがって、「物件評価に特化した」シミュレーターを作成する場合、収益還元法（直接還元法）の計算結果と並列して、積算評価額を自動算出する機能を実装することが、競合ツールとの最大の差別化要因かつ必須要件となる 14。
3. 計算ロジックの詳細設計：NOIの構築とパラメーター
直接還元法の精度は、分子である「純収益（NOI）」の算出精度に依存する。多くの簡易ツールでは「家賃収入 × 80%」といった概算を用いるが、本ツールでは不動産鑑定評価基準に準拠した精緻な積み上げ方式を採用すべきである。
3.1 総潜在収入（GPI: Gross Potential Income）
計算の出発点は、満室想定の年間収入である。
構成要素:
貸室賃料収入: 現在稼働中の部屋は契約賃料、空室部分は「市場賃料（Market Rent）」で評価する 16。※ここが重要であり、現在の募集賃料が相場より高い場合、適正な市場賃料に引き直して計算する機能（レントロール分析）があると望ましい。
共益費収入: 管理費・共益費の満室想定額。
駐車場収入: 附帯する駐車場の契約料および空き区画の市場価格。
その他収入: 自動販売機設置料、屋上アンテナ基地局使用料、太陽光売電収入、礼金・更新料等の返還を要しない一時金（平準化して計上） 17。
3.2 実効総収入（EGI: Effective Gross Income）
GPIから、空室や未回収による損失を差し引いた、現実に得られるであろう収入。

$$EGI = GPI - (\text{空室等損失} + \text{貸倒れ損失})$$
空室損（Vacancy Loss）:
入力ロジック: エリアや物件タイプに応じたデフォルト値を設定すべきである。
都心3区・築浅: 3.0% - 5.0%
地方都市・ファミリータイプ: 5.0% - 8.0%
郊外・築古・単身向け: 10.0% - 15.0% 9
シミュレーターでは、ユーザーが想定空室率をスライダー等で調整できる機能が不可欠である。
滞納損（Collection Loss）:
日本の賃貸市場では家賃保証会社の利用が一般的になっているため、実務上は0%～1.0%程度で見積もることが多いが、リスクとして計上するオプションを残すべきである。

注意点（NOIとNCFの違い）:
不動産鑑定評価基準では、上記に加え「資本的支出（CAPEX: 大規模修繕等）」を控除したものを純収益（NCF: Net Cash Flow）とし、これを還元利回りで割り戻すのが一般的である 4。
一方、投資家の簡易計算では、CAPEX控除前のNOI（Net Operating Income）を用いることが多く、用語の定義が混在している。
シミュレーターとしては、「NOI利回り（CAPEX前）」と「NCF利回り（CAPEX後）」の両方を算出・表示する仕様が最も親切であり、プロフェッショナルな仕様と言える。
4. キャップレート（還元利回り）の決定ロジックと調査手法
直接還元法において、価格を決定づける最も感度の高い変数がキャップレート（$R$）である。$R$が0.5%変われば、評価額は10%以上変動する。したがって、シミュレーターはいかにして「妥当なキャップレート」をユーザーに提案できるかが勝負となる。
4.1 キャップレートの構造と決め方
理論的に、還元利回りは以下の式で表される。

$$R = \text{リスクフリーレート} + \text{リスクプレミアム} - \text{期待成長率}$$
リスクフリーレート: 10年物国債利回り（JGB）。2025年時点では金利上昇局面（約1.0%～1.5%程度）にあり、これがベースとなる 24。
リスクプレミアム: 不動産の個別性（流動性の低さ、築古リスク、テナント信用リスク）に対する対価。
期待成長率: 賃料が上昇するならマイナス（利回り低下＝価格上昇）、下落するならプラス（利回り上昇＝価格下落）。
4.2 実務的なキャップレートの調査方法・データソース
シミュレーター開発者は、以下の公的・民間データをデータベース化し、ユーザーの入力条件（エリア・築年数・構造）に応じて適切な利回りをレコメンド（推奨表示）する必要がある。

4.3 イールドギャップ（YG）という考え方
2025年の金利上昇局面において、投資家は「イールドギャップ（実質利回り － 借入金利）」を重視している 26。
目安: FCR（Free and Clear Return: ネット利回り）とローン金利の差が**1.5% ～ 2.0%**確保できているかが投資適格ラインとされる。
シミュレーター機能として、算出したキャップレートと、ユーザーが入力した（あるいは想定される）ローン金利との差を表示し、「適正なレバレッジ効果が得られるか」を判定する機能は非常に有用である。
5. 実務での活用と銀行融資評価（ストレス金利）の組み込み
本シミュレーターが単なる計算機を超えて「投資判断ツール」となるためには、銀行の融資審査ロジック（Financial Engineering）を内包する必要がある。実務上、物件価格は「市場価格」と「銀行評価額」の低い方によってキャップされることが多いからである。
5.1 銀行融資での評価シミュレーション（ストレス金利）
銀行は実際の適用金利（例：変動1.0%）ではなく、将来の金利上昇を見越した**「ストレス金利（審査金利）」**を用いて返済比率を計算する 25。
審査ロジックのアルゴリズム
NOIの算出: 前述の厳しめのOPEXを引いたNOIを使用（レントロールの引き直し含む）。
ストレス金利の適用:
都市銀行・大手地銀: 3.0% ～ 3.5%
スルガ銀行・一部ノンバンク: 4.5% ～ 36
一般的に「固定金利 + 1.0%」または「3.5%～4.5%」のレンジでストレステストを行う 35。
返済期間の設定: 「法定耐用年数 － 経過年数」が原則。ただし、RCは適合すれば「47年－経過年数」や「50年、55年まで延長」等のバリエーションがある 39。
DSCR（借入金償還余裕率）判定:

$$DSCR = \frac{NOI}{\text{年間元利返済額（ストレス金利ベース）}}$$
このDSCRが1.2 ～ 1.3以上となる借入金額が、銀行が貸せる上限額（融資可能額）となる 40。
シミュレーターの出力機能:
「この物件の収益価格は1億円ですが、銀行評価上の融資可能額は7,500万円です。したがって、フルローンは困難で、2,500万円＋諸費用の自己資金が必要です」という具体的なアドバイスを表示することで、ユーザーの意思決定を強力にサポートできる。
5.2 物件購入判断・売却価格算定での活用
購入時: ターゲットとするCCR（自己資金配当率）から逆算して「指値（購入希望価格）」を算出する機能。
例：「自己資金1000万円でCCR10%を目指すなら、物件価格は9,200万円以下で買う必要がある」等の逆算ロジック。
売却時: 現在のNOIと、将来（5年後など）の想定市場キャップレートを用いて出口価格を予測する。
感度分析（Sensitivity Analysis）テーブルを実装し、キャップレートが0.2%上昇/下落した場合の価格変動幅をマトリクス表示すると、プロ仕様のツールとなる。

8.1 収益価格算出アルゴリズム

Python


def calculate_direct_cap_value(
    annual_rent_gpi: float,
    occupancy_rate: float,
    structure_type: str, # 'RC', 'S', 'Wood'
    building_age: int,
    elevator: bool,
    area_cap_rate_base: float
):
    # 1. 実効総収入 (EGI)
    vacancy_loss = annual_rent_gpi * (1 - occupancy_rate)
    collection_loss = annual_rent_gpi * 0.005 # 0.5% default
    egi = annual_rent_gpi - vacancy_loss - collection_loss

    # 2. 運営費用 (OPEX) 詳細積み上げ
    # PM Fee (5%)
    pm_cost = egi * 0.05
    # BM Cost (Structure dependent)
    if structure_type == 'RC':
        bm_base = 50000 # Monthly base
    else:
        bm_base = 10000
    if elevator:
        bm_base += 20000 # EV maintenance
    bm_cost = bm_base * 12
    
    # Tax (Approx 1.2 months rent equivalent for simplicity or user input)
    tax_cost = annual_rent_gpi * 0.08 
    
    # Repair Reserve (Age dependent)
    if building_age < 10:
        repair_ratio = 0.03
    elif building_age < 20:
        repair_ratio = 0.05
    else:
        repair_ratio = 0.07
    repair_cost = annual_rent_gpi * repair_ratio

    total_opex = pm_cost + bm_cost + tax_cost + repair_cost
    
    # 3. 純収益 (NOI)
    noi = egi - total_opex

    # 4. キャップレート調整
    # Age penalty
    age_penalty = 0.0
    if building_age > 20:
        age_penalty = 0.005 # +0.5%
    if structure_type == 'Wood':
        age_penalty += 0.01 # +1.0%

    final_cap_rate = area_cap_rate_base + age_penalty

    # 5. 収益価格
    price = noi / final_cap_rate
    
    return {
        "price": price,
        "noi": noi,
        "opex_ratio": total_opex / egi,
        "cap_rate": final_cap_rate
    }
