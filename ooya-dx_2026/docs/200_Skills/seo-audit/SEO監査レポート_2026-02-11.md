## 総合評価

サイト全体として**SEO基盤は強い**。ページごとのmetadata、canonical URL、構造化データ（Organization, Person, WebSite, BreadcrumbList, SoftwareApplication）、動的サイトマップ、E-E-A-Tシグナルが充実している。

ただし、**ホームページの構造化データ重複**、**壊れたSearchAction**、**OpenGraph未設定ページの多さ**など、修正すべき問題が複数存在する。

## Critical（優先度1）— 3件

### 1. ホームページでJSON-LD構造化データが重複・矛盾している

**問題**: `layout.tsx` と `page.tsx` の両方でOrganizationとWebSiteの構造化データを定義しており、ホームページに4つのJSON-LDブロックが出力される。しかも**内容が矛盾**している。

**該当ファイル**:
- `app/layout.tsx` (11〜56行目)
- `app/page.tsx` (38〜73行目)

**矛盾の詳細**:

| フィールド | layout.tsx | page.tsx |
|-----------|-----------|---------|
| Organization name | `株式会社StartupMarketing` | `大家DX` |
| sameAs | 12件のURL | **空配列 `[]`** |
| contactPoint | なし | あり（email, Japanese） |
| WebSite SearchAction | `/search?q=...` | `/media?q=...` |

**影響**: Googleが矛盾するデータを受け取り、どちらを信頼すべきか判断できなくなる。特にOrganizationのsameAsが空配列だと、layout.tsxで設定した12件の外部リンク（東京商工会議所、埼玉DXパートナー等）の信頼性シグナルが弱まる。

**修正方針**: `app/page.tsx` の38〜73行目のJSON-LD定義（`websiteJsonLd` と `organizationJsonLd`）と、80〜87行目の `<script type="application/ld+json">` タグ2つを削除する。`layout.tsx` のみにOrganization・WebSiteを残す。ただし、`layout.tsx` のOrganizationに `contactPoint` と `address` を追加する（page.tsxにはあるがlayout.tsxにない情報）。

### 2. SearchActionが存在しない `/search` ページを指している

**問題**: `layout.tsx` のWebSite構造化データで、SearchActionのtargetが `https://ooya.tech/search?q={search_term_string}` になっているが、`/search` ルートは存在しない。

**該当ファイル**: `app/layout.tsx` (48〜53行目)

```typescript
potentialAction: {
  '@type': 'SearchAction',
  target: {
    '@type': 'EntryPoint',
    urlTemplate: `${BASE_URL}/search?q={search_term_string}`,
  },
  'query-input': 'required name=search_term_string',
},
```

**影響**: Googleがサイトリンク検索ボックスを表示した場合、ユーザーがクリックすると404ページに到達する。Google側でこのURLをテストして問題を検出する可能性もある。

**修正方針**: 3つの選択肢がある：
1. **SearchActionを削除する**（最も安全。サイト内検索が不要なら推奨）
2. **`/media?q=...`に変更する**（page.tsxの方のURL。ただしmediaページが検索クエリパラメータに対応しているか確認が必要）
3. **`/search` ページを作成する**（将来的に検索機能を実装する場合）

### 3. `/media` ページにH1タグがない

**問題**: メディアトップページ（`/media`）にH1タグが存在しない。ヒーローセクションのテキストが `<p>` タグになっている。

**該当ファイル**: `app/media/page.tsx` (103〜107行目)

```tsx
<p className="text-2xl sm:text-3xl md:text-4xl text-gray-900 whitespace-nowrap">
  テック×不動産で、意思決定を数字で。
</p>
<p className="mt-3 text-base sm:text-lg text-gray-700 whitespace-nowrap">
  現役大家が賃貸経営のシミュレーションをわかりやすく解説するメディア
</p>
```

**影響**: H1はページの主題をGoogleに伝える最重要タグ。H1がないと、Googleはページの主題を正確に理解できず、「大家DXジャーナル」「賃貸経営」などの重要キーワードでの順位に影響する。

**修正方針**: ヒーローセクションの最初の `<p>` を `<h1>` に変更する。もしくは、metadataのtitle（`大家DXジャーナル｜賃貸経営の実践ノウハウ`）と一致するH1を別途追加する。

### 4. 会社サブページ・プロフィールページにOpenGraphタグがない

**問題**: 会社情報の12サブページとプロフィールページにOpenGraph（OG）タグが設定されていない。SNSでシェアされた際にタイトル・画像・説明が正しく表示されない。

**該当ファイル（OG未設定）**:
- `app/profile/page.tsx` — タイトル・description・canonicalのみ
- `app/company/service/page.tsx`
- `app/company/portfolio/page.tsx`
- `app/company/csr/page.tsx`
- `app/company/sdgs/page.tsx`
- `app/company/teambeyond/page.tsx`
- `app/company/plastics-smart/page.tsx`
- `app/company/lib-partner/page.tsx`
- `app/company/climate-adaptation/page.tsx`
- `app/company/consumer-policy/page.tsx`
- `app/company/link/page.tsx`
- `app/company/contact/layout.tsx`

**比較**: ツールページ（roi, irr等）はすべてOGタグが設定済み。

**修正方針**: 各ページのmetadataに以下を追加：

```typescript
  card: 'summary_large_image',
  title: 'ページタイトル',
  description: 'ページ説明',
  images: [`${BASE_URL}/images/media/hero-media.jpeg`],
},
```

### 5. `document.title` がサーバー側メタデータを上書きしている

**問題**: 複数のクライアントコンポーネントで `document.title` を使ってブラウザ表示のタイトルを上書きしている。これにより、**Googlebotが見るタイトル（SSR）とユーザーが見るタイトルが異なる**。

**該当ファイル（主要なもの）**:

| ファイル | SSRタイトル | document.title |
|---------|-----------|---------------|
| `app/LandingPageClient.tsx:28` | `不動産投資クラウドソフトなら「大家DX」｜現役大家が開発` | `大家DX｜賃貸経営をもっとスマートに` |
| `app/pricing/page.tsx:14` | `有料プランについて \| 大家DX` | `有料プランについて \| 大家DX`（同じ） |
| `app/faq/page.tsx:22` | `よくある質問 \| 大家DX` | `よくある質問 \| 大家DX`（同じ） |
| `app/disclaimer/page.tsx:9` | `免責事項 \| 大家DX` | `免責事項 \| 大家DX`（同じ） |

**影響**: ホームページが最も深刻。SSRタイトルの方がSEOキーワード（「不動産投資」「クラウドソフト」「現役大家」）を含んでおり、client-sideで上書きするとそのキーワードがユーザーのブラウザタブから消える。Googlebotは基本的にSSR版を見るので検索結果への影響は少ないが、不一致は望ましくない。

**修正方針**: `LandingPageClient.tsx` の28行目の `document.title = '...'` を削除する。SSRのmetadata exportで設定されたタイトルをそのまま使う。FAQ、pricing、disclaimer等はSSRと同じ値なので実害はないが、冗長なので削除しても良い。

### 6. ユーティリティページ（template-demo等）に `noindex` がない

**問題**: 開発用・テンプレート用のページがGoogleにインデックスされる可能性がある。

**該当ファイル**:
- `app/tools/template-demo/page.tsx` — テンプレートデモページ
- `app/tools/components/page.tsx` — コンポーネント一覧ページ
- `app/tools/_template/page.tsx` — テンプレート（`_`プレフィックスで自動除外される可能性あり）

**影響**: 低品質ページがインデックスされると、サイト全体の品質評価に悪影響を与える可能性がある。

**修正方針**: 各ページのmetadataに追加：

```typescript
robots: {
  index: false,
  follow: false,
},
```

また、`app/sitemap.ts` の19行目の除外リストに `template-demo` を追加：

```typescript
if (entry.name === 'components') return false;
if (entry.name === 'template-demo') return false; // 追加
```

## Quick Wins（優先度3）— 3件

### 7. タイトル区切り文字の不統一（`｜` vs `|`）

**問題**: ページタイトルのブランド名区切りに全角 `｜` と半角 `|` が混在している。

**全角 `｜` を使用しているページ（例）**:
- `不動産投資クラウドソフトなら「大家DX」｜現役大家が開発` （ホーム）
- `賃貸経営のROI（投資利益率） 計算シミュレーション｜CCR・FCR同時計算` （ROI）
- `大家DXジャーナル｜賃貸経営の実践ノウハウ` （メディア）

**半角 `|` を使用しているページ（例）**:
- `よくある質問 | 大家DX` （FAQ layout）
- `有料プランについて | 大家DX` （Pricing layout）
- `お問い合わせ | 大家DX` （Contact layout）
- `免責事項 | 大家DX` （Disclaimer layout）
- `会社概要 | 株式会社StartupMarketing` （Company）

**修正方針**: 全角 `｜` に統一する（日本語サイトでは一般的）。対象は主にlayoutファイル4件と会社ページのmetadata。

---

### 8. ToolStructuredDataに `datePublished`/`dateModified` が未設定

**問題**: ほとんどのツールページで `ToolStructuredData` コンポーネントに `datePublished` と `dateModified` を渡していない。これらが未設定だとWebPage構造化データが生成されず、Googleに公開日・更新日を伝えられない。

**現状**: `navigation.ts` に各ツールの `lastUpdated` データが存在するが、個別ページから `ToolStructuredData` に渡していない。

**ROIページの例** (`app/tools/roi/page.tsx:53-57`):
```tsx
<ToolStructuredData
  name="ROI（投資利益率）シミュレーター"
  description="賃貸経営のROI（投資利益率）を計算するツール"
  toolPath="/tools/roi"
  // datePublished と dateModified が未設定
/>
```

**修正方針**: 各ツールページで `datePublished` と `dateModified` を追加。`navigation.ts` の `publishDate` と `lastUpdated` をインポートして使用する。約30ファイルが対象。

---

### 9. sitemapから `template-demo` を除外すべき

**問題**: `app/sitemap.ts` のツール自動検出ロジックで `template-demo` ディレクトリが除外されていない。

**該当ファイル**: `app/sitemap.ts` (15〜20行目)

```typescript
.filter((entry) => {
  if (!entry.isDirectory()) return false;
  if (entry.name.startsWith('_')) return false;
  if (entry.name === 'components') return false;
  if (entry.name === 'template-demo') return false; // ← この行を追加
  const pagePath = path.join(toolsDir, entry.name, 'page.tsx');
  return fs.existsSync(pagePath);
})
```

**影響**: `template-demo` がサイトマップに含まれると、Googleがクロールしてインデックスする可能性がある。

## 長期的な改善（優先度4）— 4件

### 10. セクション別のOG画像作成

**問題**: 全ページで同じOG画像（`/images/media/hero-media.jpeg`）を使用している。SNSでシェアされた際にすべてのページが同じプレビュー画像になり、クリック率が下がる。

**修正方針**: 最低限、セクション別にOG画像を分ける：
- `/tools/*` → ツール系の画像
- `/media/*` → メディア系の画像
- `/company/*` → 会社系の画像
- ホームページ → 専用のブランド画像

将来的にはNext.jsの `opengraph-image.tsx` を使って動的OG画像生成も検討できる。

---

### 11. sitemapの `lastModified` を実際の日付にする

**問題**: `app/sitemap.ts` で静的ページ・会社ページ・ツールページのすべてが `lastModified: new Date()` を返している。ビルドのたびに「全ページが更新された」とGoogleに伝えることになり、信頼性が下がる。

**該当ファイル**: `app/sitemap.ts` (62, 69, 79, 99, 100, 138行目など)

**修正方針**:
- ツールページ: `navigation.ts` の `lastUpdated` フィールドを使用
- 静的ページ: 実際の最終更新日をハードコーディング
- 記事ページ: 既にMDXの `date` フィールドを使用しているので問題なし（36〜38行目）

---

### 12. パンくずリストの統一

**問題**: セクションごとにパンくずリストの形式が異なる。

| セクション | パンくず形式 | 区切り |
|-----------|------------|-------|
| ツール一覧 | `大家DX > 計算ツール一覧` | `ChevronRight`アイコン |
| メディア | `🏠 / 大家DXジャーナル` | `/`テキスト |
| 記事 | `大家DXジャーナル > カテゴリ` | SVGアイコン |
| プロフィール | `ホーム > 自己紹介...` | `>`テキスト |
| 会社 | `ホーム > 会社概要` | `>`テキスト |

**修正方針**: 共通のBreadcrumbコンポーネントを作成し、全セクションで統一する。JSON-LDのBreadcrumbListも常にホームページから開始するようにする（現在、メディア記事のBreadcrumbListは `/media` から始まっている）。

---

### 13. ネストされたページをsitemapに追加

**問題**: `/tools/brokerage/guide` のようなネストされたツールページがサイトマップに含まれていない。サイトマップの自動検出は `/tools/{slug}/page.tsx` のみを対象としており、`/tools/{slug}/{sub}/page.tsx` は検出されない。

**該当ファイル**: `app/sitemap.ts` (9〜29行目の `getToolSlugs` 関数)

**修正方針**: `getToolSlugs` を拡張してサブディレクトリも再帰的に検出するか、手動でネストページを追加する。

## 良好な点（変更不要）

- **E-E-A-T構造化データ**: Person → Organization のIDチェーンが適切に実装されている
- **robots.ts**: 適切なブロック設定
- **canonical URL**: 全ページに設定済み
- **301リダイレクト**: URL移行が適切に処理されている
- **ツールページのメタデータ**: ユニークなtitle、description、keywords（keywordsはGoogle非対応だが害はない）
- **フッターの内部リンク**: 全セクション・全ツールへのリンクが充実
- **記事の構造化データ**: Article、BreadcrumbList、author/publisherリファレンスが適切
